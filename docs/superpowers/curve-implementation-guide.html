들어가며: 왜 Curve 인가

머리카락, 털(fur), 잔디, 스파게티 같은 가늘고 긴 기하는 삼각형 메시로 모델링하기에는 너무 얇다. 한 가닥의 머리카락을 원기둥 메시로 만들면 수십 개의 삼각형이 필요한데, 장면에 수백만 가닥이 있다면 메모리와 BVH 가 감당하지 못한다. pbrt-v4 의 Curve shape 는 이 문제를 정면으로 해결한다. 하나의 가닥을 3차 베지에(cubic Bezier) 중심선(centerline) + 폭(width) 으로 표현하여, 메시 한 가닥당 수십 개 삼각형 대신 단 4개의 control point 와 2개의 width 값으로 끝낸다.

이 자료가 다루는 실전 맥락은 Moana Island Scene 을 caramel 로 임포트 하는 작업이다. 이 장면은 약 536만 개의 cylinder hair 곡선과 3.4만 개의 flat 곡선을 포함한다. 즉 Curve 를 제대로 포팅하지 못하면 섬의 식생과 야자수 잎을 렌더링할 수 없다. 곡선 한 개당 메모리를 1바이트라도 아끼는 것이 곧 장면 전체를 메모리에 올릴 수 있느냐를 가른다.

곡선 교차의 핵심 아이디어는 의외로 단순하다. 레이를 +z 축으로 만드는 좌표계로 곡선을 투영하면, "레이가 곡선에 맞는가"라는 3D 문제가 "곡선이 (x,y)=(0,0) 축에서 반-폭(half-width) 이내를 지나는가"라는 2D 디스크 문제로 붕괴한다. 그리고 곡선이 충분히 평평해질 때까지 재귀적으로 절반씩 쪼개는(de Casteljau subdivision) 것으로 그 2D 문제를 선형 근사로 푼다. 베지에의 두 가지 핵심 성질 — *convex hull property* (곡선은 control point 의 볼록 껍질 안에 있다) 와 *variation diminishing* (쪼갤수록 곡선이 직선에 가까워진다) — 이 이 알고리즘의 정확성을 떠받친다.

Section 1 — Curve 정의 방법

1.1 Curve 는 "일반화된 원기둥(generalized cylinder)"

pbrt 의 Curve 는 한 문장으로 요약된다: 3차 베지에 중심선을 따라 원(또는 리본)을 휩쓸어(sweep) 만든 곡면. 중심선은 4개의 control point 로 정의되는 cubic Bezier 이고, 그 위 각 점에서 중심선에 수직인 두 방향으로 width/2 만큼 양쪽으로 오프셋하면 곡면(엄밀히는 평평한 2D 리본)이 만들어진다. width 는 시작점과 끝점에서 각각 주어지고 그 사이는 선형 보간된다.

여기서 결정적인 설계 사실 하나: 이 서브시스템 전체는 내부적으로 오직 "control point 4개짜리 cubic Bezier 한 segment"만 다룬다. B-spline, 2차(quadratic) 곡선, 여러 segment 로 이어진 체인 — 이 모든 것은 장면 파싱 시점에 cubic Bezier segment 들로 변환되어 들어온다. 런타임 교차 경로(hot path)는 절대로 quadratic 이나 B-spline 을 보지 않는다. 이 단순함이 곧 성능이다.

1.2 CurveType: Flat / Cylinder / Ribbon — 왜, 어디에, 어떻게 다른가

휩쓸어 만든 리본을 "어떻게 향하게 하고 어떻게 셰이딩할지"가 CurveType (shapes.h:1195, enum class CurveType { Flat, Cylinder, Ribbon }) 으로 결정된다. 세 타입은 같은 중심선·같은 width·같은 교차 코드를 공유한다. 차이는 단 두 군데뿐이다: (a) 충돌 폭(hitWidth)을 보정하는가, (b) 셰이딩 노멀을 어떻게 만드는가. 그런데 이 두 군데가 시각적으로 완전히 다른 결과를 낳는다. 각 타입을 왜 존재하는지 / 어디에 쓰는지 / 기하와 셰이딩이 어떻게 다른지 세 축으로 본다.

caramel 은 세 타입을 모두 지원한다. caramel 은 hit 당 dpdu/dpdv 를 저장하지 않고 노멀에서 유도한 ONB(Coordinate) 하나만 저장하므로, 타입별 미분 계산은 오직 셰이딩 노멀 n 을 만들어 내기 위한 수단으로만 존재한다 (Section 2.9, 3 참조).

Flat — 레이를 향하는 빌보드

  • 왜: 가장 싸다. 화면에서 1픽셀도 안 되는 가느다란 hair/fur 는 단면이 원인지 평면인지 눈이 구분하지 못한다. 그러면 "항상 카메라(정확히는 레이)를 정면으로 바라보는 납작한 띠"로 충분하다. 단면을 레이에 수직으로 세워 두면 어느 각도에서 봐도 폭이 그대로 보이므로, 둥근 튜브의 실루엣을 공짜로 흉내 낸다.
  • 어디에: 멀리 있는 fur/hair, 화면 기여가 작은 ground cover. Moana Island 에서 34,018 개의 Flat curve 가 xgGrass, xgFronds, xgLeaves 같은 지표 식생(ground cover)에 쓰인다 — 카메라에서 멀어 평평함이 드러나지 않는 잎·풀 더미다.
  • 기하: 평평한 리본. 단면(폭 방향)이 항상 레이에 수직인 평면 안에 놓이도록 ray space 에서 폭을 펼친다.
  • 셰이딩: 노멀은 그냥 레이 반대 방향 n = -ray.d 이다 (빌보드라 폭을 가로질러도 노멀이 변하지 않는다). pbrt 에서는 dpdv = objectFromRay(Normalize(-dpduPlane.y, dpduPlane.x, 0)·hitWidth) 로 폭 벡터를 만들고 n = Cross(dpdu, dpdv) 가 되는데, 이 cross 는 ray space 의 +z(= 레이 방향)와 정렬되어 결과적으로 n ∥ -ray.d 이다. caramel 은 미분 없이 곧장 n = -ray.d 로 둘 수 있다 — 이것이 Flat 의 핵심 단순화다.

Cylinder — 평평한 기하 + 비튼 노멀로 둥근 튜브 흉내

  • 왜: hair 가 화면에서 몇 픽셀 두께로 보이기 시작하면 빌보드의 "평평함"이 드러난다 — 폭 전체가 똑같은 밝기로 셰이딩되어 종이처럼 보인다. 진짜 둥근 튜브라면 가장자리로 갈수록 노멀이 시선에서 멀어져 어두워지는 음영(self-shading gradient)이 있어야 한다. Cylinder 는 기하는 그대로 평평하게 두고 셰이딩 노멀만 폭을 가로질러 비틀어 이 음영을 만든다. 즉 비용은 Flat 과 거의 같은데 실루엣 안쪽에 둥근 입체감이 생긴다.
  • 어디에: 가까운 머리카락·털 — 평평함이 눈에 띄는 거리. Moana Island 에서 5,363,484 개의 Cylinder curve 가 동물 털/머리카락(animal hair/fur)에 쓰인다. 섬 곡선의 절대다수가 이 타입이다.
  • 기하: Flat 과 완전히 동일하다 (같은 평평한 리본, 같은 폭 펼침). 기하만 보면 Flat 과 Cylinder 는 구분할 수 없다.
  • 셰이딩: 노멀을 폭 좌표 v 에 따라 접선축 둘레로 회전시킨다. 충돌점에서 θ = Lerp(v, -90°, +90°) 만큼 폭 벡터를 접선(dpdu) 둘레로 돌린 뒤 n = Cross(dpdu, dpdv) 를 취하면, 노멀이 한쪽 가장자리(v=0)에서 -90°, 중심선(v=0.5)에서 정면, 반대 가장자리(v=1)에서 +90° 로 반-원기둥(half-cylinder) 단면의 노멀을 정확히 재현한다. 왜 ± 90° 가 반-원기둥을 만드는지는 Section 2.9 에서 유도한다. caramel 은 이를 위해 dpdu(전역 접선)와 ray-space 폭 벡터가 필요하다.

Ribbon — 진짜로 방향이 있는 비틀린 띠

  • 왜: 잎사귀나 잔디 날은 실제로 납작한 면이고 고유한 앞/뒷면(authored normal)이 있다. 이들을 빌보드로 두면 카메라가 옆에서 봐도 정면 폭으로 보여 잎이 항상 마주 보는 부자연스러움이 생기고, Cylinder 로 두면 둥글게 음영져 잎이 튜브처럼 보인다. Ribbon 은 양 끝점에 명시적 노멀 n[0], n[1] 을 주고 그 사이를 구면 선형 보간(slerp)하여, 곡선이 길이를 따라 비틀리는(예: 바람에 꼬인 잎) 진짜 방향성 있는 면을 만든다. 레이마다 다시 방향을 잡지 않는 유일한 타입이라 물리적으로 일관적이다.
  • 어디에: 명시적 노멀이 authored 된 잎·꽃잎·잔디. Moana Island 자체에는 Ribbon 이 0개다(섬 식생은 멀어서 Flat/Cylinder 로 충분). 그러나 잎/풀을 노멀까지 정확히 표현하려는 일반적 용도에서는 Ribbon 이 물리적으로 올바른 유일한 선택이다 — caramel 이 잎·풀을 제대로 셰이딩하려면 이 타입이 필요하다.
  • 기하: 폭 방향이 끝점 노멀로 정의되는 평면 안에 놓이며, 그 평면이 u 를 따라 비틀린다. Flat/Cylinder 와 달리 폭 방향이 레이에 의존하지 않는다. 대신 리본이 레이에 비스듬히 서면 겉보기 폭이 줄어드는 것을 반영해 hitWidth 를 단축(foreshorten)한다: hitWidth *= |nhit · ray.d| / |ray.d|.
  • 셰이딩: 노멀은 보간된 nhit 자체가 아니라 그것으로 만든 폭 벡터에서 나온다 — dpdv = Normalize(Cross(nhit, dpdu))·hitWidth, 그리고 n = Cross(dpdu, dpdv). 결과적으로 셰이딩 노멀은 nhit 방향과 일치한다(폭을 가로질러 일정). slerp 공식의 유도는 Section 2.9.

1.3 정의 데이터: CurveCommon 과 Curve

데이터는 두 구조로 나뉜다. 하나의 논리적(그려진) 곡선의 모든 segment 가 공유하는 데이터와, 각 segment 가 개별로 갖는 데이터다.

CurveCommon (shapes.h:1200, 공유 데이터):

  • CurveType type;
  • Point3f cpObj[4]; — 4개의 cubic Bezier control point. object space 에 그대로 저장된다(render space 로 변환하지 않음; 생성자 shapes.cpp:471 의 cpObj[i] = c[i]). 항상 정확히 4개(CHECK_EQ(c.size(), 4)).
  • Float width[2]; — u=0, u=1 에서의 폭. 사이는 선형 보간
  • Normal3f n[2]; — Ribbon 의 양끝 노멀(생성자에서 Normalize)
  • Float normalAngle, invSinNormalAngle; — AngleBetween(n[0], n[1]) 와 1/sin 값. slerp 용 미리 계산값(Ribbon 전용)
  • const Transform *renderFromObject, *objectFromRender; — 변환 쌍(서로 역행렬). cpObj 가 object space 이므로 교차 시 레이를 object space 로 끌어오고, hit 을 다시 render space 로 보낼 때 쓴다.
  • bool reverseOrientation, transformSwapsHandedness; — 후자는 renderFromObject->SwapsHandedness() 로 생성자에서 캐시된다.

Curve (shapes.h:1219, 실제 Shape) 는 놀랄 만큼 작다:

  • const CurveCommon *common; — 공유 데이터 포인터
  • Float uMin, uMax; — 이 Curve 인스턴스가 담당하는 [0,1] 의 부분 구간

즉 Curve = (공유 중심선) + (u 부분 구간). 이것이 메모리 트릭의 핵심이다. 한 논리적 곡선의 N 개 segment 인스턴스가 단 하나의 CurveCommon 을 공유하고, 서로는 (uMin, uMax) 만 다르다. 536만 개의 hair 가 메모리에 들어가는 이유가 바로 이것이다.

장면 파일은 다음을 준다: P(control point 배열), degree(2 또는 3, 기본 3), basis("bezier" 또는 "bspline", 기본 bezier), width/width0/width1, type, N(ribbon 노멀), splitdepth(기본 3).

segment 개수 결정 (shapes.cpp 의 실제 검사식):

  • bezier basis: (nP-1-degree) mod degree = 0 이어야 하고, nSegments = (nP-1)/degree. segment 들은 끝점 하나를 공유하며 겹친다 — 첫 segment 는 degree+1 개 점을 쓰고, 이후 각 segment 는 직전 segment 의 마지막 점 + 새 점 degree 개를 재사용한다. control point 오프셋(cpOffset)이 segment 마다 degree 씩 전진.
  • bspline basis: nP ≥ degree+1 이어야 하고, nSegments = nP - degree. degree+1 개 점의 슬라이딩 윈도우가 한 칸씩(cpOffset += 1) 전진한다. 윈도우가 겹치는 것이 바로 B-spline 을 C2 연속으로 만드는 이유다.

각 segment 마다 4-점 cubic Bezier segCpBezier 를 만든다:

  • bezier+degree3: 4점 그대로 복사
  • bezier+degree2: 3점에 ElevateQuadraticBezierToCubic
  • bspline+degree3: 4점 윈도우에 CubicBSplineToBezier
  • bspline+degree2: 3점 윈도우에 QuadraticBSplineToBezier 후 ElevateQuadraticBezierToCubic

width 도 함께 쪼갠다: 이 segment 의 양끝 폭 = Lerp(seg/nSegments, width0, width1) 와 Lerp((seg+1)/nSegments, width0, width1).

Ribbon 노멀의 쪼갬: N 은 nSegments+1 개여야 하고(끝점마다 하나), segment seg 는 n[seg], n[seg+1] 두 개를 받는다.

Level B — splitdepth 에 의한 기하 사전 분할 (CreateCurve, shapes.cpp:494)

cubic Bezier segment 하나가 주어지면 2splitdepth 개(기본 splitdepth=3 → 8개)의 Curve 인스턴스를 만든다. 각각 uMin = i/n, uMax = (i+1)/n 이고 모두 같은 CurveCommon 을 가리킨다. 이것은 빌드 시점에 한 번 수행되는 정적·균일 u 분할로, BVH 가 더 작은(타이트한) leaf 박스를 갖게 하고 런타임 재귀가 더 작은 호(arc)에서 시작하도록 한다. CreateCurve 는 기하를 다시 분할하지 않는다 — u 도메인만 자른다. cpObj 는 전체 4 control point 그대로이고, 각 인스턴스는 필요할 때 CubicBezierControlPoints(cpObj, uMin, uMax) 로 자기 부분 호를 평가한다. (GPU 경로에서는 splitdepth=0 으로 두어 분할을 GPU dicing 에 맡긴다.)

정리: 그려진 곡선 → (Level A) M 개 cubic Bezier segment → (Level B) M × 2splitdepth 개 Curve Shape 인스턴스가 BVH 에 들어간다.

두 종류의 "depth" 를 혼동하지 말 것. splitdepth (파싱 시점, 기본 3 → 곡선당 8개 고정 segment, 각각 BVH leaf shape)는 maxDepth (레이마다, segment 마다 평탄도 휴리스틱으로 정해지는 재귀 깊이, Section 2.8)와 전혀 다르다. caramel 은 둘 다 필요하다 — BVH 친화적 정적 segment 와 런타임 적응 재귀.

2.1 Cubic Bezier 의 기본: Bernstein 기저와 그 성질의 유도

4개 control point p0, p1, p2, p3 로 정의되는 cubic Bezier 곡선은 (책 식 6.16):

p(u) = (1-u)3 p0 + 3(1-u)2 u p1 + 3(1-u) u2 p2 + u3 p3, u ∈ [0,1].

계수들은 Bernstein 기저 다항식 Bin(u) 다. degree n, 인덱스 i 에 대해 이항계수 C(n,i) = n!i! (n-i)! 를 써서

Bin(u) = C(n,i) ui (1-u)n-i, i = 0,…,n.

cubic(n=3)이면 B03=(1-u)3, B13=3(1-u)2u, B23=3(1-u)u2, B33=u3 로 위 식의 네 항과 정확히 일치한다.

(성질 1) 단위 분할(partition of unity). 이항정리에 의해

∑i=0n Bin(u) = ∑i=0n C(n,i) ui(1-u)n-i = (u + (1-u))n = 1n = 1.

(성질 2) 비음성(non-negativity). u ∈ [0,1] 이면 u ≥ 0, 1-u ≥ 0, C(n,i) > 0 이므로 Bin(u) ≥ 0.

(성질 3) 볼록 껍질(convex hull). 성질 1+2 에 의해 임의의 u ∈ [0,1] 에서 p(u) 는 control point 들의 볼록 결합(convex combination) — 계수가 음 아니고 합이 1 — 이다. 볼록 결합은 정의상 점들의 볼록 껍질 안에 있다. 따라서 곡선 전체가 control point 의 볼록 껍질 안에 놓인다. 이것이 뒤의 모든 bounding box 컬링(2.5)의 정당성이다: control point 만 감싸면 곡선은 자동으로 포함된다.

보간 성질: u=0 이면 B03(0)=1, 나머지 0 → p(0)=p0. u=1 이면 B33(1)=1 → p(1)=p3. 즉 곡선은 p0, p3 를 통과하지만 p1, p2 는 통과하지 않는다 — 이 둘은 형상을 "끌어당기는" 역할만 한다.

(성질 4) variation diminishing 의 직관. 임의의 직선(또는 평면)과 control polygon 의 교차 횟수가 k 이면, 곡선과 그 직선의 교차 횟수는 k 이하다. 즉 곡선은 control polygon 보다 "덜 구불거린다." 이 성질의 정량적 형태(쪼갤수록 현에 가까워지는 4-d 수렴)는 2.5, 2.8 에서 쓴다.

미분(접선): Bernstein 기저의 미분 공식 dduBin = n(Bi-1n-1 - Bin-1) 을 대입해 정리하면 degree 가 하나 내려간 Bezier 형태가 된다:

p'(u) = 3 ∑i=02 Bi2(u) (pi+1 - pi) = 3[(1-u)2(p1-p0) + 2(1-u)u(p2-p1) + u2(p3-p2)].

즉 "degree-n Bezier 의 미분 = n 곱하기, 1차 차분 (pi+1-pi) 을 control point 로 하는 degree-(n-1) Bezier." 이 형태가 2.3 의 de Casteljau 미분 규칙과 2.8 의 2차 차분 곡률 측정에 직접 연결된다.

2차 미분: 위를 한 번 더 미분하면 (2.8 에서 결정적으로 쓰임)

p''(u) = 6[(1-u) Δ2 p0 + u Δ2 p1], Δ2 pi = pi+2 - 2pi+1 + pi,

즉 2차 차분 Δ2 pi 를 control point 로 하는 degree-1 Bezier 에 n(n-1)=6 을 곱한 것이다. 따라서 maxu |p''(u)| ≤ 6max(|Δ2 p0|, |Δ2 p1|) — 이 부등식이 2.8 의 상수 6의 출처다.

2.2 Blossoming (극형식, polar form): 모든 마법의 근원

cubic 의 blossom f(u0, u1, u2) 는 다음 세 조건을 만족하는 함수다:

  1. 대각 성질(diagonal): f(u,u,u) = p(u).
  2. 대칭(symmetric): 인자 순서를 바꿔도 같다, f(u0,u1,u2)=f(uσ(0),uσ(1),uσ(2)).
  3. 다중 아핀(multi-affine): 각 인자에 대해 따로 보면 아핀(1차)이다, 예컨대 f(Lerp(t,a,b), u1,u2) = Lerp(t, f(a,u1,u2), f(b,u1,u2)).

유일성: degree-n 다항식에 대해 이 세 조건을 만족하는 n-변수 blossom 은 유일하다(blossoming principle). 이유: 다중 아핀+대칭이면 f 는 각 변수의 1차 단항식들의 대칭합으로 전개되고, 그 계수들은 대각 성질 f(u,…,u)=p(u) 가 곧 p 의 계수를 못 박으므로 완전히 결정된다. 이 유일성 덕분에 "blossom 으로 계산한 control point" 는 표현 방식과 무관하게 같은 곡선을 준다 — 이 사실이 2.6 의 변환들을 정당화한다.

계산은 de Casteljau 인데 각 레벨에서 보간 매개변수가 다르다 (splines.h:18, BlossomCubicBezier):

ak = Lerp(u0, pk, pk+1), k=0,1,2bj = Lerp(u1, aj, aj+1), j=0,1f(u0,u1,u2) = Lerp(u2, b0, b1).

레벨 1 이 u0, 레벨 2 가 u1, 레벨 3 이 u2 를 쓴다. 세 인자가 모두 같으면(u0=u1=u2=u) 보통의 de Casteljau(2.3)로 환원된다.

왜 중요한가 — 구간 제한과 blossom 값: 동일한 cubic 을 구간 [a,b] 로 제한했을 때의 4개 Bezier control point 가 정확히 다음 blossom 값이다:

f(a,a,a), f(a,a,b), f(a,b,b), f(b,b,b) .

왜 하필 이 네 값인가 (유도): 새 곡선 q(s) = p(Lerp(s,a,b)), s∈[0,1] 의 Bezier control point qi 를 구하고 싶다. 일반 정리에 의해 degree-3 Bezier 의 i 번째 control point 는 "세 인자 중 i 개를 끝값 b 로, 나머지 3-i 개를 시작값 a 로 둔 blossom" 이다. 즉 q0=f(a,a,a), q1=f(a,a,b), q2=f(a,b,b), q3=f(b,b,b). 직관: control point q0 은 s=0(전부 a)에서의 위치라 f(a,a,a)=p(a), q3 은 s=1(전부 b)이라 f(b,b,b)=p(b) 로 끝점을 정확히 맞춘다. 중간 두 점 f(a,a,b), f(a,b,b) 는 다중 아핀성 덕분에 끝점 접선까지 정확히 재현하도록 자동으로 결정된다. 이로써 위 네 값이 나온다. ∎

이 단 하나의 사실이 (1) 부분 호 추출(CubicBezierControlPoints)과 (2) B-spline → Bezier 변환을 동시에 떠받친다. Curve(uMin, uMax) 인스턴스가 자기 u 부분 구간 + 공유 cpObj 로부터 구체적인 4-점 cubic 을 얻는 방법이 바로 이것이다 (splines.h:56):

CP_[umin, umax] = f(umin,umin,umin), f(umin,umin,umax), f(umin,umax,umax), f(umax,umax,umax) .

임의의 [umin, umax] (절반만이 아닌)에 작동하므로, 단순 절반 분할이 아니라 blossom 이 필요하다.

2.3 평가와 미분: de Casteljau

값만 필요할 때 EvaluateCubicBezier(u) = f(u,u,u) — 평범한 de Casteljau (splines.h:26):

level 1: c1k = Lerp(u, pk, pk+1), k=0,1,2level 2: c2j = Lerp(u, c1j, c1j+1), j=0,1point: p(u) = Lerp(u, c20, c21).

미분이 필요하면 같은 nested Lerp 를 레벨 2까지 돌린 뒤 (splines.h:35):

p'(u) = 3 (c21 - c20).

왜 이게 맞는가 (유도): 레벨-2 점은 그 자체가 끝점이 c1 인 1차 보간이다. de Casteljau 의 마지막 단계는 p(u)=Lerp(u,c20,c21)=(1-u)c20+u c21. 그런데 c20, c21 는 degree-1 Bezier(즉 직선 구간)의 끝점이고, 일반 미분 공식 p'(u) = n∑ Bin-1(u)(pi+1-pi) 에서 n=3 일 때 마지막 비축약 단계가 정확히 3(c21 - c20) 를 준다. 계수 3은 degree n 이다 — "degree-n Bezier 의 미분 = n 곱하기 마지막에서 두 번째 de Casteljau 점들의 차." 차수 n 일반화: p'(u) = n (cn-11 - cn-10).

영-접선 폴백(load-bearing). c21 = c20 (control point 가 겹쳐 접선이 0) 이면 p'(u) = p3 - p0 (현, chord) 로 폴백한다 (splines.h:40, LengthSquared(cp2[1]-cp2[0]) > 0 검사). dpdu 가 절대 영벡터가 되어선 안 되기 때문이다(RecursiveIntersect 가 CHECK_NE(Vector3f(0,0,0), dpdu), shapes.cpp:691). dpdu 가 0 이면 Cross(dpdu,dpdv) 노멀이 0 이 되어 정규화에서 NaN 이 난다. 현으로 폴백하는 이유: 접선이 국소적으로 0 이어도 곡선의 전체 진행 방향은 p3 - p0 이 가장 합리적인 대용이고, 절대 0 이 아니다(겹친 끝점이 아닌 한). 값-전용 평가기를 포팅하고 미분을 따로 계산할 때 이 폴백을 빠뜨리면 dpdv/노멀에서 NaN 이 난다.

2.4 de Casteljau 중점 분할 (Subdivision)

u=0.5 에서 cubic 을 두 개의 cubic 으로 쪼개면, u=0.5 가 모든 Lerp 를 단순 평균 a+b2 로 만들어 닫힌 형식(closed form)이 된다 (splines.h:45). 7개 점을 반환하고 두 자식이 가운데 점 q3 를 공유한다:

q0 = p0 q1 = p0+p12 q2 = p0+2p1+p24q3 = p0+3p1+3p2+p38 q4 = p1+2p2+p34 q5 = p2+p32q6 = p3.

유도 (왼쪽 자식, u=1/2 de Casteljau 를 전개): c10=p0+p12, c11=p1+p22, c12=p2+p32. 레벨 2: c20=c10+c112=p0+2p1+p24, c21=c11+c122=p1+2p2+p34. 레벨 3: p(frac12)=c20+c212=p0+3p1+3p2+p38. 왼쪽 자식의 Bezier control point 는 de Casteljau 의 "왼쪽 가장자리" 값들 p0, c10, c20, p(frac12) = q0,q1,q2,q3 이고, 오른쪽은 "오른쪽 가장자리" p(frac12), c21, c12, p3 = q3,q4,q5,q6 다. ∎

가중치가 이항계수 (1,1)/2, (1,2,1)/4, (1,3,3,1)/8 — 즉 C(k,·)/2k — 임에 주목. RecursiveIntersect 가 u 방향으로 segment 를 이등분할 때 쓴다. (검증 팁: q3 = EvaluateCubicBezier(cp, 0.5) 인지 확인.)

2.5 Bounding: convex hull 과 variation diminishing

BoundCubicBezier(cp) = Union(AABB(p0, p1), AABB(p2, p3)).

곡선이 control point 의 볼록 껍질 안에 있으므로(2.1 성질 3) 이 박스는 보수적(곡선을 반드시 포함)이다. 4개 점을 모두 묶는 대신 두 개의 segment-box AABB(p0,p1), AABB(p2,p3) 를 union 하는 것은 — 네 점을 다 포함하므로 — 동등하면서 약간 더 싸다. 부분 구간 버전(splines.h:68)은 먼저 CubicBezierControlPoints 로 부분 호의 CP 를 뽑은 뒤 그것을 묶으므로 훨씬 타이트한 박스를 준다(umin=0,umax=1 이면 전체 버전으로 단락).

Variation diminishing property (정량 버전): 핵심 교육 포인트는 "AABB 는 구불구불한 곡선을 타이트하게 감싸지 못하지만, Bezier 를 쪼개면 덜 구불거리게 된다." 이를 정량화하면: 곡선과 그 현(chord, p0 → p3) 사이의 최대 거리는 2차 차분으로 한계 지어지고, 한 번 이등분할 때마다 그 한계가 1/4 로 준다. 유도는 다음과 같다.

(a) 현으로부터의 편차 ≤ (1/8)·max 2차 미분. 표준 결과로, 구간 길이 1 위에서 함수 g 와 그 끝점을 잇는 직선(현) 사이의 최대 편차는 18max|g''| 이하다(보간 오차 정리; 1차 보간의 나머지항 g''(ξ)2s(1-s) 의 최대값이 s=1/2 에서 18max|g''|). Bezier 의 각 좌표 성분에 적용하면

‖p - chord‖∞, 성분당 ≤ 18maxu |p''(u)|.

(b) max 2차 미분 ≤ 6 L0. 2.1 에서 p''(u)=6[(1-u)Δ2 p0 + uΔ2 p1] 이므로 (성분당) maxu|p''| ≤ 6max(|Δ2 p0|,|Δ2 p1|) =: 6 L0.

(c) 이등분 시 2차 차분이 1/4. 2.4 의 왼쪽 자식에 대해

Δ2 q0 = q0 - 2q1 + q2 = p0 - (p0+p1) + p0+2p1+p24 = p0 - 2p1 + p24 = 14 Δ2 p0.

(오른쪽 자식·Δ2 q1 도 같은 식으로 frac14Δ2 p1.) 따라서 한 번 이등분하면 L0 → L0/4, d 번이면 L0/4d. 즉 현으로부터의 편차는 d 회 이등분 후 ∼ L0/4d 로 기하급수적으로 작아진다. ∎

이 (a)(b)(c)가 (1) 쪼갤수록 BVH 박스가 현에 밀착해 타이트해지고 (2) 재귀 교차(2.9)에서 leaf 를 직선(현)으로 근사하는 것을 정당화하며 (3) 2.8 의 깊이 공식을 그대로 만든다.

2.6 Degree elevation 과 B-spline → Bezier 변환

이 변환들은 모두 2.2 의 blossom 유일성에서 나온다 — "같은 곡선의 blossom 값은 표현 무관하게 같다."

Degree elevation (quadratic Q0, Q1, Q2 → cubic, splines.h:76):

Q0, Lerp(23, Q0, Q1), Lerp(13, Q1, Q2), Q2 .

유도: degree 를 n 에서 n+1 로 올릴 때 새 control point 는 Q'i = in+1Qi-1 + (1-in+1)Qi (표준 차수 상승 공식). n=2 → n+1=3:

  • i=0: Q'0 = Q0.
  • i=1: Q'1 = frac13 Q0 + frac23 Q1 = Lerp(frac23, Q0, Q1) (Lerp 의 t=frac23 이 Q1 쪽 가중치 frac23).
  • i=2: Q'2 = frac23 Q1 + frac13 Q2 = Lerp(frac13, Q1, Q2).
  • i=3: Q'3 = Q2.

같은 quadratic 을 cubic 으로 정확히(근사가 아니라) 표현한다 — 두 곡선이 모든 u 에서 동일.

Quadratic B-spline → Bezier (균일 노트 위 점 p01, p12, p23, splines.h:81):

Lerp(12, p01, p12), p12, Lerp(12, p12, p23) .

코드 주석의 라벨링(p01,p12,p23 → p11,p12,p22)대로, 균일 노트에서 한 span 의 Bezier 끝점은 인접 B-spline 점의 중점이고 가운데는 그대로다. frac12 는 균일 노트 간격에서 양옆 blossom 인자가 대칭이라 나온다.

Cubic B-spline → Bezier (균일 노트 위 점 p012, p123, p234, p345, splines.h:93):

p122 = Lerp(23, p012, p123), p223 = Lerp(13, p123, p234),p233 = Lerp(23, p123, p234), p334 = Lerp(13, p234, p345),결과 = Lerp(12, p122, p223), p223, p233, Lerp(12, p233, p334) .

1/3, 2/3, 1/2 의 출처: B-spline 의 한 control point 는 blossom 으로 pi,i+1,i+2=f(ti,ti+1,ti+2) (연속한 세 노트에서의 blossom 값)다. 우리가 원하는 Bezier 점은 현재 span [t2,t3] 의 끝값 a=t2, b=t3 로 평가한 f(a,a,a),f(a,a,b),f(a,b,b),f(b,b,b) (2.2). 균일 노트 ti = i 로 두면 각 목표 blossom 값을 주어진 p 들로부터 다중 아핀성으로 풀어 쓸 때 보간 비율이 정확히 1/3, 2/3(노트 세 칸 중 한 칸·두 칸 위치)과 1/2(대칭 평균)로 떨어진다. 이것이 코드의 Lerp 상수다. (참고 그림: 코드 주석의 berkeley cubicbsplinepoints.)

이 변환들은 전부 파싱 시점(Curve::Create)에서 끝낸다 — hot path 는 단일 4-점 cubic 만 본다.

2.7 Ray-space LookAt 투영: 핵심 트릭

이제 교차의 심장부다. 목표는 레이를 +z 축으로 만드는 좌표계로 곡선을 옮기는 것. 그러면 레이 원점 → (0,0,0), 레이 방향 → +z 가 되어, "레이가 곡선에 맞는가"가 2D 문제로 붕괴한다: "곡선 위 점의 x2+y2 ≤ (w/2)2 가 어떤 z ∈ [0, |ray.d|· tmax] 에서 성립하는가." x, y 는 레이로부터의 수직 거리, z 는 레이를 따라간 거리다. 반복되는 leaf 테스트가 전부 2D 점/선/디스크 + 1D z-범위 테스트로 단순해지므로 연산이 대폭 줄어든다.

up 벡터 선택과 왜 dx = ray.d × chord 가 최적인가 (책 그림 6.33): bounding box 는 ray space 의 x,y 축에 정렬된 AABB 다. 곡선 전체가 대략 현 p0 → p3 방향으로 뻗어 있으므로, 현을 한 축(여기선 x축)에 맞추면 곡선이 그 축을 따라 길고 수직 축으로는 얇아져 AABB 가 가장 타이트해진다. ray space 에서 z 는 레이 방향으로 고정되어 있으니, 남은 자유도는 x,y 평면의 회전뿐 — 이를 현이 x축에 오도록 고정한다. LookAt 의 up 벡터를 dx = ray.d × (p3 - p0) 로 주면, LookAt 내부에서 right = Normalize(up × dir) 가 만들어지고 dir=Normalize(ray.d) 이므로

right = Normalize((ray.d×chord)×ray.d).

벡터 삼중곱 (a× b)× a = |a|2 b - (a· b)a 에 의해 이는 chord 에서 ray.d 성분을 제거한 것 — 즉 chord 의 레이수직 평면으로의 사영이다. 따라서 right(=x축)가 곡선이 뻗는 방향(투영된 현)과 정렬되어 AABB 가 최소가 된다. dx 자체는 ray.d 와 chord 둘 다에 수직이라 곧 ray space 의 y축이 된다.

레이가 현과 평행하면 ray.d×chord=0, |dx|2 = 0 이 되어 특이(singular)해지므로, 이때는 CoordinateSystem(ray.d) 로 임의의 수직 기저를 잡는다 (Duff et al. 의 branchless ONB). 현이 한 점으로 보이는 경우라 어느 회전을 잡아도 무방하다.

rayFromObject = LookAt(ray.o, ray.o + ray.d, dx), cpi = rayFromObject(cpObji).

pbrt 의 LookAt(pos, look, up) (transform.cpp:81)은 다음 열(column)을 갖는 worldFromCamera 를 만든다: dir=Normalize(look-pos), right=Normalize(Cross(Normalize(up),dir)), newUp=Cross(dir,right), 그리고 열 순서 (right, newUp, dir, pos). 반환은 Transform(cameraFromWorld, worldFromCamera) — 즉 그 객체의 forward 변환은 cameraFromWorld 다. forward 를 점에 적용하면 object → ray space 로 간다. 여기서 dir = Normalize(ray.d) 가 세 번째(+z) 기저이므로 z 가 레이를 따라 증가하고, ray space 에서 ray.o ↦ (0,0,0), ray.d ↦ (0,0,|ray.d|) 가 된다.

caramel 에는 LookAt 이 없다. 직접 만들어야 한다(Section 3.4). pbrt 는 left-handed look 을 쓴다(right = Normalize(up × dir), 이 순서, transform.cpp). cross product 순서를 정확히 복제하지 않으면 x/y 가 거울 반전되고, 이는 edgeFunc(2.9 L5)를 통해 v 부호를 뒤집을 뿐 아니라 현 기반 up 벡터 처리를 어긋나게 한다. 검증법: 만든 변환으로 ray.o 가 원점, ray.o+ray.d 가 (0,0,|ray.d|) 로 가는지, p3 - p0 가 대략 +x 를 향하는지 확인.

2.8 재귀 깊이 휴리스틱: 얼마나 쪼갤 것인가

교차 전에 "이 호를 몇 번 이등분할지"를 정한다. 책의 표현: "가장 미세한 정제 수준에서 선형화된 곡선과의 최대 거리가 작은 고정 거리보다 작도록" 충분히 쪼갠다.

curviness 측정 = control point 의 2차 차분(second difference) 의 최대 성분 (ray-space CP, shapes.cpp:586):

L0 = max_i ∈ 0,1 max( |Δ2 cpi.x|, |Δ2 cpi.y|, |Δ2 cpi.z| ), Δ2 cpi = cpi - 2 cpi+1 + cpi+2.

2.1·2.5 에서 보였듯 2차 차분은 곡선이 직선(현)에서 벗어나는 정도를 한계 짓고(p''=6·[2차 차분의 degree-1 Bezier]), cubic 의 현으로부터 최대 편차가 이 값의 상수배로 묶인다. ray-space 에서 측정하므로 width 와 같은 단위다.

eps = 0.05 · max(w0, w1) (= width/20), maxDepth = Clamp( 12 Log2Int( √2· 6 · L08 · eps ), 0, 10 ).

L0 ≤ 0 (직선)이면 maxDepth =0.

완전한 유도 (상수 √2·6/8 포함): 목표는 "d 번 이등분 후 현으로부터의 편차 ≤ eps" 인 최소 d. 2.5 의 결과들을 합치면, d 회 이등분한 가장 미세한 sub-curve 의 (성분당) 현 편차는

편차(d) ≤ 18 maxu|p''(u)| · 14d ≤ 6 L08·14d.

  • 6 : max|p''| ≤ 6 L0 (2.1, 2.5b — cubic 의 n(n-1)=6).
  • 14d : 이등분 1회마다 2차 차분 ×frac14 (2.5c).

x,y 두 성분을 따로 묶은 AABB 의 대각선 여유를 보태기 위해 √2 를 곱한다(평면 거리 √ex2+ey2 ≤ √2 max(ex,ey)). 따라서 안전한 편차 한계는

편차(d) ≤ √2· 6 L08·14d.

이를 ≤ eps 로 만들려면

√2·6 L08· 4d ≤ eps ⇒ 4d ≥ √2·6 L08 eps ⇒ d ≥ log4√2·6 L08 eps = 12log2√2·6 L08 eps.

이것이 코드의 frac12Log2Int(√2·6 L0/(8 eps)) 와 정확히 일치한다. 나누기 2가 log2 를 log4 로 바꾼다. 상수 √2·6/8 ≈ 1.0607 은 (b)(a)의 6/8 에 (x,y 평면) √2 안전계수를 접어 넣은 것이다. tolerance 가 width 에 비례하는 이유: 곡선 폭의 1/20 보다 훨씬 작은 오차는 눈에 보이지 않으므로 더 쪼갤 필요가 없다. [0,10] 클램프로 최악의 경우 작업량을 제한한다(210 leaf).

pbrt 의 Log2Int(float) 는 반올림(round-to-nearest) 이지 floor 가 아니다 (math.h:372, significand 를 21.5 의 significand 와 비교해 +1). L0/eps 가 4의 거듭제곱 경계 바로 아래일 때 floor 기반보다 깊이가 1 더 나올 수 있다(미관상 차이, 정확성 무관). caramel 이 std::log2+floor 를 쓰면 가장자리 케이스에서 깊이가 ~1 다를 수 있다.

2.9 재귀 교차: 분기와 leaf

RecursiveIntersect 는 두 모드다. rayLength = Length(ray.d) 를 매 호출마다 다시 구하는데, 여기서 cp 는 ray space 지만 전달되는 ray 변수는 object-space 레이임에 주의 — z 비교가 rayLength 로 단위 변환을 한다(ray space 의 z 는 |ray.d| 단위, object t 는 1 단위).

depth > 0 (분기): SubdivideCubicBezier(cp) 로 7점/두 자식을 만들고 u = u0, (u0+u1)/2, u1. 각 자식에 대해 maxWidth(양끝 u 의 width 최대)로 박스를 0.5·maxWidth 확장하여 rayBounds 와 겹침 검사(겹치지 않으면 continue), depth-1 로 재귀. shadow ray(si==nullptr)면 첫 hit 에서 즉시 true 반환(any-hit 빠른 종료).

depth == 0 (leaf, 실제 테스트): ray-space (x,y) 평면의 2D 테스트 + z 범위 검사. 순서대로:

(L1) 끝점 edge 테스트 — segment 끝 너머의 점을 기각 (이웃 segment 가 담당해야 함). 각 끝점에서 접선을 법선으로 하는 반평면에 대한 2D edge function:

estart = (cp1.y - cp0.y)(-cp0.y) + cp0.x(cp0.x - cp1.x) ≥ 0,
eend = (cp2.y - cp3.y)(-cp3.y) + cp3.x(cp3.x - cp2.x) ≥ 0.

유도/해석: 시작점에서 접선 방향은 t = cp1 - cp0. 시작 캡을 지나며 법선이 이 접선인 평면을 생각하면, edge function 은 t 와 시작 캡에서 원점으로의 벡터 -cp0 의 내적이다: estart = (cp1 - cp0)·(-cp0) = (cp1.x-cp0.x)(-cp0.x) + (cp1.y-cp0.y)(-cp0.y). pbrt 가 쓴 cp0.x(cp0.x-cp1.x) 항은 (cp1.x-cp0.x)(-cp0.x) 와 같다. estart ≥ 0 이면 원점이 시작 캡 평면의 “앞쪽”(접선이 가리키는 segment 안쪽)에 있다는 뜻이라 통과, 아니면 이웃 segment 의 영역이라 기각한다. eend 는 접선을 뒤집어(cp2 - cp3, segment 안쪽을 향함) 같은 식으로 본다: eend = (cp2 - cp3)·(-cp3). 이렇게 segment 경계에서 이중 카운트/누락을 막는다.

(L2) 현에 원점 투영 → 매개변수 w (점-선 투영, 책 그림 6.37):

segDir = cp3.xy - cp0.xy, w = (-cp0.xy) · segDir|segDir|2.

유도: 현을 직선 ℓ(w) = cp0.xy + w·segDir 로 보고 원점 (0,0) 에 가장 가까운 w 를 구한다. ddw|ℓ(w)|2 = 0 ⇒ 2 segDir·(cp0.xy + w segDir)=0 ⇒ w = -cp0.xy·segDir|segDir|2. 이것이 원점의 현 위로의 정사영 매개변수다. |segDir|2 = 0 (퇴화: 현이 한 점)이면 기각.

(L3) u 와 hitWidth:

u = Clamp(Lerp(w, u0, u1), u0, u1), hitWidth = Lerp(u, width0full, width1full).

hitWidth 는 전역 width[] 를 전역 u 로 평가한다(width 는 논리적 곡선 전체에 대한 단일 선형 함수이므로 common->width[0..1] 에 전역 u 를 넣는다 — segment-local 폭이 아님).

Ribbon 조정 — u 에서 노멀 slerp 후 폭을 단축(foreshorten):

nhit = sin((1-u)θ) n0 + sin(uθ) n1sinθ, θ = normalAngle, hitWidth *= |nhit · ray.d||ray.d|.

slerp 공식의 유도: 단위 노멀 n0, n1 사이 각이 θ = ∠(n0,n1) 일 때, 구면 선형 보간은 n(u) 를 두 벡터가 펼치는 평면 안에서 n0 로부터 각 uθ 만큼, n1 로부터 각 (1-u)θ 만큼 떨어지도록 정의한다. n(u) = α n0 + β n1 로 놓고 두 조건 n(u)· n0 = cos(uθ), n(u)· n1 = cos((1-u)θ) 를 n0· n1=cosθ 와 함께 풀면

α + βcosθ = cos(uθ), αcosθ + β = cos((1-u)θ).

이 2×2 선형계의 해는 α = sin((1-u)θ)sinθ, β = sin(uθ)sinθ (삼각 항등식 cos(uθ) - cos((1-u)θ)cosθ = sin((1-u)θ)sinθ 사용). 따라서

nhit = n(u) = sin((1-u)θ)sinθ n0 + sin(uθ)sinθ n1,

코드의 sin0 = sin((1-u)θ)·invSinNormalAngle, sin1 = sin(uθ)·invSinNormalAngle 와 정확히 일치한다. θ = 0 (두 노멀 동일, sinθ=0 으로 나눌 수 없음)이면 nhit = n0 로 단락한다(shapes.cpp:659). 단축(foreshorten)의 이유: 리본 면이 레이에 비스듬할수록 화면 투영 폭이 줄어든다. 단위 레이 방향을 d = ray.d/|ray.d| 라 하면 겉보기 폭 비율은 면 노멀과 레이가 이루는 각의 코사인 = |nhit· d| 이므로 hitWidth 에 이를 곱한다(코드에서는 |nhit·ray.d|/|ray.d|).

(L4) 곡선 점 평가 + 반-폭 디스크 테스트:

dpdu = EvaluateCubicBezier(common→cpObj, u, \&dpdu).

전체 곡선의 control point 를 전역 u 로 평가한다(segment 의 cp 가 아니라!). segment 의 local cp + local w 로 평가하면 chain-rule 인자 (umax-umin) 만큼 크기가 틀린 접선이 나온다.

로 만들어진다(필요시 reverseOrientation ^ transformSwapsHandedness 로 뒤집음). caramel 은 dpdu/dpdv 를 저장하지 않고 노멀 1개 + 그로부터 만든 ONB(Coordinate) 만 저장하므로, 위 미분 작업은 오직 이 한 줄의 n 을 얻기 위해 존재한다.

  • Flat: dpdu 는 곡선 접선, dpdv 는 ray space 의 (x,y) 수직을 object 로 되돌린 것 → Cross(dpdu,dpdv) 가 ray space 의 +z(레이 방향)와 정렬되어 n = -ray.d 가 된다. 그래서 caramel 은 Flat 을 미분 없이 곧장 n=-ray.d 로 둘 수 있다.
  • Ribbon: n = Cross(dpdu, Normalize(Cross(nhit,dpdu))·hitWidth). Cross(dpdu, Cross(nhit,dpdu)) = |dpdu|2 nhit - (dpdu· nhit)dpdu — 삼중곱이 nhit 에서 접선 성분을 뺀 것이므로, 셰이딩 노멀이 nhit 방향과 일치한다(폭을 가로질러 일정). caramel 은 dpdu 와 nhit 만으로 노멀을 얻는다.
  • Cylinder: 아래에서 따로 유도.

Cylinder twist 의 유도 — 왜 θc = Lerp(v,-90°,+90°) 가 반-원기둥 노멀인가. ray space 에서 곡선을 반지름 R = hitWidth/2 인 원기둥으로 본다고 하자. 단위 접선을 t = dpduPlane 방향, 그에 수직인 단위 폭 방향을 w = dpdvPlane 방향이라 두자. 진짜 원기둥의 표면점은 단면각 φ 로 매개화되며 표면 노멀은 단면 안에서 n(φ) = sinφ w + cosφ (t× w) 처럼 접선축 둘레로 도는 벡터다. 우리는 평평한 리본 위 폭 좌표 v∈[0,1] 를 원기둥 둘레의 보이는 반쪽(앞면, φ∈[-90°,+90°])에 사상하려 한다 — v=0(한 가장자리)→φ=-90°, v=0.5(중심선)→φ=0(정면), v=1(반대 가장자리)→φ=+90°. 이 선형 사상이 정확히 φ = θc = Lerp(v,-90°,+90°) 다.

코드는 이를 노멀 대신 폭 벡터 dpdvPlane 을 접선 dpduPlane 둘레로 -θc 회전시켜 구현한다: Rotate(-θc, dpduPlane). 회전된 dpdv 와 dpdu 의 cross 를 취하면 노멀이 접선축 둘레로 θc 만큼 돈 셈이 되어, 가장자리(v=0,1)에서는 노멀이 시선(+z)에 대해 ±90° 기울어(거의 옆을 향해) 어둡게, 중심선(v=0.5)에서는 정면을 향해 밝게 셰이딩된다. 즉 평평한 기하 위에 둥근 튜브의 음영 그라디언트가 생긴다. 부호(-θc)는 회전 방향과 cross 순서를 맞춰 v 가 증가할 때 노멀이 올바른 쪽으로 쓸리도록 한다. 최종 노멀 n=Cross(dpdu,dpdv) 는 이 회전된 dpdv 를 그대로 받는다.

마지막으로 hit 을 SurfaceInteraction 으로 만들고 renderFromObject 로 render space 에 되돌린다. flipNormal = reverseOrientation ^ transformSwapsHandedness. wo 로는 -ray.d(object space) 를 넘긴다.

2.10 매개화 요약

  • u ∈ [0,1]: 중심선을 따라(논리적 곡선 전역). 각 Curve 인스턴스는 [umin, umax] 를 담당. leaf 가 현 매개 w 를 u = Clamp(Lerp(w, u0, u1), u0, u1) 로 사상.
  • v ∈ [0,1]: 폭을 가로질러. 중심선에서 v=0.5, v = 0.5 ± ptCurveDist/hitWidth (부호는 edgeFunc).
  • dpdu: object space 곡선 접선 (EvaluateCubicBezier(cpObj, u), 전역 u).
  • dpdv: 폭 방향 × hitWidth, CurveType 별. Cylinder 는 추가로 dpdu 둘레 θc = Lerp(v, -90°, +90°) 회전.
  • 노멀: 모든 타입이 n = normalize(Cross(dpdu, dpdv)). Flat 은 -ray.d 로 단순화, Ribbon 은 nhit 방향, Cylinder 는 폭 가로질러 비틀림. caramel 은 이 n 하나만 저장(ONB 로 확장).
  • width(u) = Lerp(u, width0, width1) (선형 taper).

Section 3 — caramel 구현 방법

3.1 클래스를 어디에 둘 것인가

Curve 는 메시가 아닌 leaf Shape 다 — Triangle 과 같은 부류이지 Instance(변환 래퍼) 가 아니다. (참고: pbrt 의 Curve 는 엄밀히는 : public Shape 서브클래스가 아니라 tagged-pointer Shape variant 라 : public Shape 로 grep 해도 안 나온다 — 여기서 "leaf Shape" 는 caramel 에서의 모델이다.) 다만 Instance 의 "레이를 local space 로 보내고 노멀을 변환하는 패턴"은 그대로 복사할 좋은 템플릿이다.

class Curve final : public Shape 를 include/shape.h 의 Triangle 옆(자유 함수 블록 L218 전)에 선언하고, 구현은 src/shapes/curve.cpp 에 둔다. 추가로 struct CurveCommon (공유 cpObj/width/type/normals)을 두어 다중 segment 곡선이 하나의 CurveCommon 을 공유하는 N 개 Curve Shape 가 되게 한다 — pbrt 를 그대로 미러링하며 각 Shape 를 단일 Bezier segment(저렴한 AABB + BVH leaf)로 유지한다. caramel 의 BVH 는 vector<const Shape*> 위에 구성되므로(scene.cpp:100), 각 segment-Curve 가 TLAS 의 독립 leaf 가 된다. 이것이 자연스러운 단위다: 하나의 Curve Shape == [umin, umax] 를 가진 하나의 cubic Bezier segment.

3.4 caramel 에 없는 것: LookAt 을 직접 만들기

caramel 에는 LookAt 이 없다(grep 확인). 4×4 행렬을 만들고 역행렬을 구할 필요 없이, dot product 로 ONB 투영하는 것이 가장 싸고 caramel 의 Coordinate 관용구와 맞다.

ray-space 기저를 만든다:

  • forward (z) = ray.m\d (이미 단위)
  • right (x) = normalize(cross(dx, ray.m\d)), where dx = cross(ray.m\d, p3 - p0); dx 가 0 이면 Coordinate{ray.m_d}.m_axis1
  • up (y) = cross(ray.m\d, right)

Coordinate 의 멤버 (m\axis1, m\axis2, m\world\n) = (right, up, forward) 로 채우면:

  • world → ray space (점): ray_frame.to_local(p - ray.m_o) — (x,y,z) 의 z 가 레이를 따른 거리.
  • ray space → world (역 = ONB 의 전치): ray_frame.to_world(v). Flat/Cylinder 의 dpdv 를 world 로 되돌릴 때만 필요.

4×4 inverse 없이 dot product 만으로 끝난다. (벡터를 ray space 로 보낼 때는 to_local 에 점이 아닌 벡터를 넣되 - ray.m_o 를 빼지 않는다.)

3.5 교차 알고리즘 포팅 (단계별)

STEP A (object space 변환): control point 를 world space 로 저장하면 이 단계를 건너뛴다. caramel 의 큰 단순화 — 곡선은 배치된 후 쓸모 있는 "object space" 가 없다. (대안: to_world 를 저장하고 Instance 처럼 transform_point/transform_vector + k 재조정 — 비추천.)

STEP B (ray-space 기저): 3.4절. 4개 cp 를 ray_frame.to_local(cp_obj[i] - ray.m_o) 로 투영.

STEP C (컬링): maxWidth = max(Lerp(umin, w0, w1), Lerp(umax, w0, w1)). curveBounds = Union(Bounds(cp0, cp1), Bounds(cp2, cp3)) 를 0.5 · maxWidth 로 (x, y, z 3축 모두) 확장. rayBounds = (0,0,0) 에서 (0,0,maxt) (|ray.d|=1). AABB::is_overlap 으로 겹침 검사, 안 겹치면 miss.

STEP D (재귀 깊이): Section 2.8 식. L0, eps = 0.05 · max(w0, w1), maxDepth = Clamp(·, 0, 10). caramel 에는 Log2Int 가 없으니 std::ilogb 또는 작은 헬퍼를 쓰고 L0 > 0 을 먼저 가드한다.

STEP E (재귀): recursive_intersect. depth>0 이면 subdivide_bezier → 7점/2 자식, 각 자식 박스를 rayBounds 와 겹침 검사 후 depth-1 재귀. depth==0 이면 leaf 테스트(Section 2.9 의 L1~L5).

한 곡선 내 closest-hit: Triangle 과 달리 한 Curve 의 재귀가 여러 leaf 에서 서로 다른 t 의 hit 를 찾을 수 있다. 직접 최소 t 를 추적해야 한다 — best.t = maxt 로 시작, 더 작은 t 만 유지. 외부 BVH 는 서로 다른 shape 끼리만 t 를 비교하지, 한 shape 내부 segment 끼리는 비교하지 않는다.

3.8 watertightness 와 splitDepth 스케일링

caramel 의 Triangle 은 공유 모서리 누수를 막기 위해 기본적으로 watertight 교차(shape.cpp:79, JCGT 2013)를 쓴다. 곡선은 공유 모서리 watertightness 문제가 없다 — 각 곡선은 고립된 리본/튜브이고, 한 곡선의 segment 들은 정확한 모서리 공유가 아니라 반-폭 디스크의 설계상 겹침으로 join 에서 만난다. 따라서 Curve 포팅은 watertight 로직이 필요 없다. 곡선의 견고함은 재귀 깊이 한계(eps = width/20)가 선형화 오차를 sub-pixel 로 보장하는 데서 온다.

splitDepth 권장: 런타임 RecursiveIntersect 가 깊이를 적응적으로 정하므로 Level B 정적 분할을 생략(splitdepth=0)해도 정확성은 유지된다 — Level B 는 BVH leaf 타이트함에만 기여한다. 다만 BVH 품질을 위해 작은 splitdepth(예: 2~3)는 유지 권장. 536만 hair 의 Moana 장면에서 splitdepth 가 메모리(× 2d 개 Shape)와 BVH 트래버설 비용을 정면으로 좌우하므로, 장면 규모에 맞춰 튜닝해야 한다.

3.9 정밀도

caramel 은 단정밀도 Float=float(common.h:32). pbrt 곡선 코드도 알고리즘상 float-stable 하다 — 2차 차분 깊이 휴리스틱과 eps=width/20 tolerance, 그리고 클램프들이 fp 오차를 흡수한다. 전부 Float 로 유지하고, L0 > 0 / denom == 0 / LengthSquared == 0 가드를 정확히 보존한다. EPSILON = 1e-3(common.h:55)은 재귀 레이 원점 오프셋(rayintersectinfo.cpp:38)에 쓰여 곡선에도 자동 적용되므로 자기교차(self-intersection) 방지에 별도 작업 불필요.

3.10 장면 와이어링

scene_parser.cpp 의 parse_shape(L226)에 else if (type == "curve") 분기를 추가하고, JSON 필드 P/width/width0/width1/degree/basis/type/N/to_world 를 파싱한다. (Ribbon 의 N 은 정점당 노멀 배열로 nSegments+1 개이며, segment 마다 양끝 노멀 n[seg], n[seg+1] 2개를 슬라이스해 그 segment 의 CurveCommon 에 넣는다.) B-spline→Bezier 변환과 segment 분할은 생성 시점에 수행하고(pbrt Curve::Create 처럼), parse_instanced_shapes(L162)가 여러 Instance 를 push 하는 선례를 따라 Bezier segment 당 하나의 Curve Shape 를 shapes 벡터에 emit 한다(각자 CurveCommon 공유). control point 는 생성 시 to_world 를 한 번 적용해 world space 로 저장하면 ray_intersect 가 레이마다 변환할 필요가 없다. 파싱 후 Scene::add_mesh_and_arealight(scene.cpp:53)가 Shape 마다 호출되는데, Curve 는 is_light()==false 라 기하로만 추가되고 build_accel(L100)이 BVH 에 넣는다 — scene.h/scene.cpp 는 손댈 것이 없다.

    // CurveType: 휩쓸린 리본을 어떻게 향하게/셰이딩할지 결정.
    //   Flat     : 리본이 레이를 향해 빌보드 (가장 저렴; 먼 거리 hair/fur)
    //   Cylinder : Flat 기하지만 셰이딩 노멀을 폭 방향으로 비틀어 둥근 튜브처럼 (근접 hair)
    //   Ribbon   : 양끝 노멀이 주어지고 u 를 따라 slerp 되는 진짜 방향 있는 띠 (잔디, 잎)
    enum class CurveType { Flat, Cylinder, Ribbon };

    // 하나의 논리적(그려진) 곡선의 모든 segment 가 공유하는 데이터. 다중 segment
    // 곡선은 N 개의 Curve shape 가 되어 모두 하나의 CurveCommon 을 가리키고 서로는
    // [u_min, u_max] 부분 구간만 다르다 -- 수백만 가닥을 메모리에 담는 핵심 트릭.
    struct CurveCommon {
        CurveCommon(const std::array<Vector3f, 4> &cp, Float w0, Float w1,
                    CurveType type, const Vector3f *n /* Ribbon 외에는 nullptr */);

        CurveType m_type;
        std::array<Vector3f, 4> m_cp;   // 4개 cubic Bezier control point, WORLD space
        Float m_width[2];               // u=0, u=1 에서의 폭 (선형 taper)

        // Ribbon 전용; Flat/Cylinder 는 기본값으로 둠.
        Vector3f m_n[2];                       // 양끝 노멀 (생성자에서 정규화)
        Float m_normal_angle = Float0;         // AngleBetween(n0, n1)
        Float m_inv_sin_normal_angle = Float0; // 1 / sin(normal_angle), slerp 용
    };

    class Curve final : public Shape {
    public:
        // 하나의 Curve == [u_min, u_max] 로 제한된 하나의 cubic Bezier segment.
        // arealight 는 항상 nullptr: 곡선은 절대 area light 가 될 수 없다.
        Curve(std::shared_ptr<const CurveCommon> common, Float u_min, Float u_max, BSDF *bsdf);

        std::pair<bool, RayIntersectInfo> ray_intersect(const Ray &ray, Float maxt) const override;
        AABB get_aabb() const override;
        Float get_area() const override;

        // --- 네 개의 stub: 곡선은 emitter 로 샘플링 불가 -----------------------
        std::tuple<Vector3f, Vector3f, Float> sample_point(Sampler &sampler) const override;
        Float pdf_solidangle(const Vector3f &, const Vector3f &, const Vector3f &) const override;
        bool is_solid_angle_sampling_possible() const override { return false; }
        const std::vector<Vector3f>& get_polygon_vertices() const override { return m_empty; }

    private:
        // depth>0 -> 분할; depth==0 -> 실제 leaf 레이/곡선 테스트.
        // cp_ray 는 이미 ray space 로 투영된 4개 control point.
        // best 는 지금까지의 가장 가까운 hit (best.t 는 maxt 로 시작).
        bool recursive_intersect(const Ray &ray, Float maxt,
                                 const std::array<Vector3f, 4> &cp_ray,
                                 const Coordinate &ray_frame,
                                 Float u0, Float u1, int depth,
                                 RayIntersectInfo &best) const;

        std::shared_ptr<const CurveCommon> m_common;
        Float m_u_min, m_u_max;
        std::vector<Vector3f> m_empty;  // get_polygon_vertices() 가 반환
    };

}

namespace Caramel {
    namespace {
        // pbrt 규약: lerp(t, a, b) = (1-t)a + t b  (t 가 첫 인자).
        template <typename T>
        T lerp(Float t, const T &a, const T &b) { return a * (Float1 - t) + b * t; }

        // Blossom f(u0,u1,u2): 레벨마다 다른 매개변수로 de Casteljau.
        Vector3f blossom(const std::array<Vector3f, 4> &p, Float u0, Float u1, Float u2) {
            const Vector3f a0 = lerp(u0, p[0], p[1]);
            const Vector3f a1 = lerp(u0, p[1], p[2]);
            const Vector3f a2 = lerp(u0, p[2], p[3]);
            const Vector3f b0 = lerp(u1, a0, a1);
            const Vector3f b1 = lerp(u1, a1, a2);
            return lerp(u2, b0, b1);
        }

        // [u_min, u_max] 로 제한한 곡선의 4개 Bezier control point.
        std::array<Vector3f, 4> sub_control_points(const std::array<Vector3f, 4> &cp,
                                                   Float u_min, Float u_max) {
            return {blossom(cp, u_min, u_min, u_min), blossom(cp, u_min, u_min, u_max),
                    blossom(cp, u_min, u_max, u_max), blossom(cp, u_max, u_max, u_max)};
        }

        // 점 + 접선 (de Casteljau). 영-접선일 때 현으로 폴백하는 것이 load-bearing:
        // dpdu 는 절대 영벡터가 되어선 안 된다.
        Vector3f eval_bezier(const std::array<Vector3f, 4> &cp, Float u, Vector3f *deriv) {
            const Vector3f a0 = lerp(u, cp[0], cp[1]);
            const Vector3f a1 = lerp(u, cp[1], cp[2]);
            const Vector3f a2 = lerp(u, cp[2], cp[3]);
            const Vector3f b0 = lerp(u, a0, a1);
            const Vector3f b1 = lerp(u, a1, a2);
            if (deriv) {
                *deriv = (b1 - b0).length() > Float0 ? (b1 - b0) * static_cast<Float>(3)
                                                     : cp[3] - cp[0];
            }
            return lerp(u, b0, b1);
        }

        // u=0.5 에서 가운데 점을 공유하는 두 cubic 으로 분할 (7개 점).
        std::array<Vector3f, 7> subdivide_bezier(const std::array<Vector3f, 4> &c) {
            return {c[0],
                    (c[0] + c[1]) * Float0_5,
                    (c[0] + c[1] * Float2 + c[2]) * static_cast<Float>(0.25),
                    (c[0] + c[1] * static_cast<Float>(3) + c[2] * static_cast<Float>(3) + c[3]) * static_cast<Float>(0.125),
                    (c[1] + c[2] * Float2 + c[3]) * static_cast<Float>(0.25),
                    (c[2] + c[3]) * Float0_5,
                    c[3]};
        }

        // BoundCubicBezier 의 절반 (convex hull property).
        AABB bound_two(const Vector3f &a, const Vector3f &b) { return AABB{a, b}; }

        // pbrt Log2Int(float) 미러: round-to-nearest (floor 아님).
        // caramel 에 Log2Int 가 없어 std::ilogb 로 근사. L0>0 가드 후에만 호출할 것.
        Float int_log2_round(Float v) {
            if (v <= Float0) return Float0;
            const int lo = std::ilogb(v);
            // 1.5*2^lo 가 경계: v 가 그 위면 lo+1 로 반올림.
            return static_cast<Float>(v >= static_cast<Float>(1.5) * std::ldexp(Float1, lo) ? lo + 1 : lo);
        }
    } // namespace

    CurveCommon::CurveCommon(const std::array<Vector3f, 4> &cp, Float w0, Float w1,
                             CurveType type, const Vector3f *n)
        : m_type{type}, m_cp{cp}, m_width{w0, w1} {
        if (type == CurveType::Ribbon && n) {
            m_n[0] = n[0].normalize();
            m_n[1] = n[1].normalize();
            using std::acos; using std::sin;
            m_normal_angle = acos(std::clamp(m_n[0].dot(m_n[1]), -Float1, Float1));
            m_inv_sin_normal_angle = Float1 / sin(m_normal_angle);
        }
    }

    Curve::Curve(std::shared_ptr<const CurveCommon> common, Float u_min, Float u_max, BSDF *bsdf)
        : Shape{bsdf, nullptr}, m_common{std::move(common)}, m_u_min{u_min}, m_u_max{u_max} {}

    AABB Curve::get_aabb() const {
        const auto cp = sub_control_points(m_common->m_cp, m_u_min, m_u_max);
        AABB box = AABB::merge(bound_two(cp[0], cp[1]), bound_two(cp[2], cp[3]));
        const Float w0 = lerp(m_u_min, m_common->m_width[0], m_common->m_width[1]);
        const Float w1 = lerp(m_u_max, m_common->m_width[0], m_common->m_width[1]);
        const Float r = std::max(w0, w1) * Float0_5;  // 폭의 절반(반지름)만큼 확장
        return AABB{box.m_min - Vector3f{r, r, r}, box.m_max + Vector3f{r, r, r}};
    }

    Float Curve::get_area() const {
        // 광원이 아니라 실제로는 dead code 지만 순수 가상이라 정의 필수.
        const auto cp = sub_control_points(m_common->m_cp, m_u_min, m_u_max);
        const Float len = (cp[1] - cp[0]).length() + (cp[2] - cp[1]).length() + (cp[3] - cp[2]).length();
        const Float avg_w = (m_common->m_width[0] + m_common->m_width[1]) * Float0_5;
        return len * avg_w;
    }

    std::tuple<Vector3f, Vector3f, Float> Curve::sample_point(Sampler &) const {
        return {vec3f_zero, Vector3f{Float0, Float0, Float1}, Float0};  // 호출 안 됨: 곡선은 광원 불가
    }
    Float Curve::pdf_solidangle(const Vector3f &, const Vector3f &, const Vector3f &) const {
        return Float0;
    }

    std::pair<bool, RayIntersectInfo> Curve::ray_intersect(const Ray &ray, Float maxt) const {
        // control point 를 WORLD space 로 저장 => object-space 변환 불필요 (pbrt STEP A 생략).
        const auto cp_obj = sub_control_points(m_common->m_cp, m_u_min, m_u_max);

        // --- ray-space 기저: +z 가 ray.d, 원점이 ray.o (pbrt LookAt 대체) ----
        // up 힌트 = 레이 방향과 p0->p3 현 둘 다에 수직 (가장 타이트한 bbox);
        // 레이가 현과 평행하면 임의 ONB 로 폴백.
        Vector3f dx = Vector3f::cross(ray.m_d, cp_obj[3] - cp_obj[0]);
        if (dx.dot(dx) == Float0) dx = Coordinate{ray.m_d}.m_axis1;
        // ray_frame.{m_axis1,m_axis2,m_world_n} = {right, up, forward};
        // to_local 이 world VECTOR 를 ray space 로, to_world 가 되돌린다 (역 == 전치).
        Coordinate ray_frame;
        ray_frame.m_world_n = ray.m_d;                                    // forward = +z
        ray_frame.m_axis1  = Vector3f::cross(dx, ray.m_d).normalize();    // right    = +x
        ray_frame.m_axis2  = Vector3f::cross(ray.m_d, ray_frame.m_axis1); // up       = +y

        std::array<Vector3f, 4> cp;
        for (int i = 0; i < 4; ++i) cp[i] = ray_frame.to_local(cp_obj[i] - ray.m_o);

        // --- 조기 컬링: 레이(원점..maxt, +z 축)와 곡선 박스 겹침 검사 ----------
        const Float w0 = lerp(m_u_min, m_common->m_width[0], m_common->m_width[1]);
        const Float w1 = lerp(m_u_max, m_common->m_width[0], m_common->m_width[1]);
        const Float r = std::max(w0, w1) * Float0_5;
        AABB cb = AABB::merge(bound_two(cp[0], cp[1]), bound_two(cp[2], cp[3]));
        cb = AABB{cb.m_min - Vector3f{r, r, r}, cb.m_max + Vector3f{r, r, r}};
        const AABB ray_box{Vector3f{Float0, Float0, Float0}, Vector3f{Float0, Float0, maxt}};
        if (!cb.is_overlap(ray_box)) return {false, RayIntersectInfo()};

        // --- 2차 차분("구불거림")으로 재귀 깊이 결정 --------------------------
        Float L0 = Float0;
        for (int i = 0; i < 2; ++i)
            for (int a = 0; a < 3; ++a)
                L0 = std::max(L0, std::abs(cp[i][a] - Float2 * cp[i + 1][a] + cp[i + 2][a]));
        int max_depth = 0;
        if (L0 > Float0) {
            const Float eps = std::max(w0, w1) * static_cast<Float>(0.05);  // width/20
            const Float d = int_log2_round(static_cast<Float>(1.41421356237) * static_cast<Float>(6) * L0 /
                                           (static_cast<Float>(8) * eps)) * Float0_5;  // /2 = log4 변환
            max_depth = std::clamp(static_cast<int>(d), 0, 10);
        }

        RayIntersectInfo best;
        best.t = maxt;
        const bool hit = recursive_intersect(ray, maxt, cp, ray_frame, m_u_min, m_u_max, max_depth, best);
        return hit ? std::pair{true, best} : std::pair{false, RayIntersectInfo()};
    }

    bool Curve::recursive_intersect(const Ray &ray, Float maxt,
                                    const std::array<Vector3f, 4> &cp,
                                    const Coordinate &ray_frame,
                                    Float u0, Float u1, int depth,
                                    RayIntersectInfo &best) const {
        if (depth > 0) {
            // --- 분기: u=0.5 에서 이등분, 각 자식 박스를 컬링 후 재귀 ----------
            const auto s = subdivide_bezier(cp);
            const Float um = (u0 + u1) * Float0_5;
            const Float ub[3] = {u0, um, u1};
            bool any = false;
            for (int seg = 0; seg < 2; ++seg) {
                const std::array<Vector3f, 4> child{s[3 * seg], s[3 * seg + 1], s[3 * seg + 2], s[3 * seg + 3]};
                const Float cw0 = lerp(ub[seg],     m_common->m_width[0], m_common->m_width[1]);
                const Float cw1 = lerp(ub[seg + 1], m_common->m_width[0], m_common->m_width[1]);
                const Float r = std::max(cw0, cw1) * Float0_5;
                AABB cb = AABB::merge(bound_two(child[0], child[1]), bound_two(child[2], child[3]));
                cb = AABB{cb.m_min - Vector3f{r, r, r}, cb.m_max + Vector3f{r, r, r}};
                const AABB ray_box{Vector3f{Float0, Float0, Float0}, Vector3f{Float0, Float0, best.t}};
                if (!cb.is_overlap(ray_box)) continue;
                any |= recursive_intersect(ray, maxt, child, ray_frame, ub[seg], ub[seg + 1], depth - 1, best);
            }
            return any;
        }

        // ---- LEAF: ray-space (x,y) 평면의 2D 테스트 + z 범위 검사 ------------
        // (L1) 양끝 캡 너머의 점 기각 (edge function). 이웃 segment 가 담당.
        if ((cp[1][1] - cp[0][1]) * (-cp[0][1]) + cp[0][0] * (cp[0][0] - cp[1][0]) < Float0) return false;
        if ((cp[2][1] - cp[3][1]) * (-cp[3][1]) + cp[3][0] * (cp[3][0] - cp[2][0]) < Float0) return false;

        // (L2) 원점 (0,0) 을 현에 투영 -> 매개변수 w.
        const Vector2f seg_dir{cp[3][0] - cp[0][0], cp[3][1] - cp[0][1]};
        const Float denom = seg_dir.dot(seg_dir);
        if (denom == Float0) return false;
        const Float w = (Vector2f{-cp[0][0], -cp[0][1]}).dot(seg_dir) / denom;

        // (L3) 전역 u + hit 시 폭 (전체 곡선 width 함수 사용).
        const Float u = std::clamp(lerp(w, u0, u1), u0, u1);
        Float hit_width = lerp(u, m_common->m_width[0], m_common->m_width[1]);

        // (L4-Ribbon) 노멀 slerp + 폭 단축(foreshorten).
        Vector3f n_hit;
        if (m_common->m_type == CurveType::Ribbon) {
            using std::sin;
            if (m_common->m_normal_angle == Float0) n_hit = m_common->m_n[0];
            else {
                const Float A = m_common->m_normal_angle, is = m_common->m_inv_sin_normal_angle;
                n_hit = m_common->m_n[0] * (sin((Float1 - u) * A) * is) + m_common->m_n[1] * (sin(u * A) * is);
            }
            hit_width *= std::abs(n_hit.dot(ray.m_d));  // caramel 에서 |ray.m_d| == 1
        }

        // (L4) 실제 곡선 점 평가 + 반-폭 디스크 테스트 (sqrt 회피 위해 제곱 비교).
        Vector3f dpcdw;
        const Vector3f pc = eval_bezier(cp, std::clamp(w, Float0, Float1), &dpcdw);
        const Float dist2 = pc[0] * pc[0] + pc[1] * pc[1];
        if (dist2 > hit_width * hit_width * static_cast<Float>(0.25)) return false;  // (hitWidth/2)^2
        if (pc[2] < Float0 || pc[2] > maxt) return false;   // rayLength == 1

        const Float t_hit = pc[2];          // pc.z / rayLength, rayLength == 1
        if (t_hit > best.t) return false;   // 이 곡선 내 가장 가까운 hit 유지

        // (L5) hit 빌드. v 는 폭을 가로질러, dpdu/dpdv 는 CurveType 별.
        const Float dist = std::sqrt(dist2);
        const Float edge = dpcdw[0] * (-pc[1]) + pc[0] * dpcdw[1];
        const Float v = edge > Float0 ? Float0_5 + dist / hit_width : Float0_5 - dist / hit_width;

        Vector3f dpdu;  // 전체 곡선 접선을 전역 u 로 평가 (leaf cp 가 아니라!)
        eval_bezier(m_common->m_cp, u, &dpdu);

        Vector3f dpdv;
        if (m_common->m_type == CurveType::Ribbon) {
            dpdv = Vector3f::cross(n_hit, dpdu).normalize() * hit_width;
        } else {
            const Vector3f dpdu_plane = ray_frame.to_local(dpdu);  // 접선을 ray space 로
            Vector3f dpdv_plane = Vector3f{-dpdu_plane[1], dpdu_plane[0], Float0}.normalize() * hit_width;
            if (m_common->m_type == CurveType::Cylinder) {
                // 폭 벡터를 dpdu_plane 둘레로 -theta 회전 -> 노멀이 폭을 가로질러 -90..+90 휩쓺.
                const Float theta = deg_to_rad(lerp(v, static_cast<Float>(-90), static_cast<Float>(90)));
                const Vector3f axis = dpdu_plane.normalize();
                using std::cos; using std::sin;
                const Float c = cos(-theta), si = sin(-theta);
                // Rodrigues 회전 (axis 둘레로 dpdv_plane 회전).
                dpdv_plane = dpdv_plane * c
                           + Vector3f::cross(axis, dpdv_plane) * si
                           + axis * (axis.dot(dpdv_plane) * (Float1 - c));
            }
            dpdv = ray_frame.to_world(dpdv_plane);  // world 로 되돌림
        }

        // caramel 은 노멀-ONB 만 저장: (dpdu, dpdv) 를 단일 노멀로 축소.
        Vector3f n = (m_common->m_type == CurveType::Flat)
                         ? ray.m_d * static_cast<Float>(-1)            // Flat 은 항상 레이를 향함
                         : Vector3f::cross(dpdu, dpdv).normalize();
        if (n.dot(ray.m_d) > Float0) n = n * static_cast<Float>(-1);   // 양면: 레이 쪽으로

        best.t = t_hit;
        best.p = ray.m_o + ray.m_d * t_hit;     // m_d 가 단위라 정확
        best.tex_uv = Vector2f{u, v};
        best.sh_coord = Coordinate{n};
        // best.shape 은 nullptr 그대로: 장면 BVH 가 이 Curve* 로 덮어쓴다.
        return true;
    }
